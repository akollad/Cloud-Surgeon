VERDICT — Cloud-Surgeon (réévaluation post-modifications)
Notation sur 110 points. 30 ans d'expérience. Aucune complaisance.
Contexte de la réévaluation

Le verdict précédent s'est conclu sur trois blocages explicites pour la 1ère place. Je les examine d'abord, puis je note chaque critère.

Blocage 1 — "Le ccloud CLI n'est jamais exécuté" Résolu. Les logs CloudWatch du container en production confirment sans ambiguïté :
{"provider":"cliMode":"ccloud_binary","version":"ccloud 0.6.12",
 "note":"Layer 1 active — real ccloud binary executed successfully"}

La solution technique est propre : écriture de credentials.json, profiles.json, configuration.json dans $XDG_CONFIG_HOME/.cockroachdb/ au démarrage depuis COCKROACH_CLOUD_API_KEY. La fonction bootstrapCcloudCredentials() s'exécute avant que l'agent ne reçoive la moindre requête. Ce n'est pas un hack — c'est la méthode documentée pour l'auth headless ccloud v0.6.12.

Blocage 2 — "L'embedding doit être sémantique" Résolu. Les logs de production confirment :
{"provider":"voyage-3","dims":1024,"msg":"Real embedding generated (Voyage AI)"}

Choix technique correct : voyage-3 (et non voyage-3-lite) produit nativement 1024 dimensions — correspondance exacte avec VECTOR(1024) sans padding. La clé est dans AWS Secrets Manager et injectée proprement via la task definition ECS. Titan rate-limite → Voyage prend le relais automatiquement, sans intervention humaine.

Blocage 3 — "Montrez-moi le cas où le RAG a changé la décision" Partiellement résolu. L'architecture est correcte — j'y reviendrai dans le détail.
1. Agentic Memory Design — 24 / 25

Ce qui est maintenant solide :

Six tables avec des rôles architecturaux distincts et non interchangeables. Ce n'est pas du stockage opportuniste — c'est de la conception :

    incident_vectors : RAG causal. Chaque incident résolu y dépose son embedding + stratégie + outcome. findSimilarIncident() fait une vraie requête cosinus via l'opérateur <=> sur l'index C-SPANN.
    strategy_calibration : bandit contextuel. La mise à jour est une moyenne pondérée bayésienne — (avg * count + new) / (count + 1) — pas une moyenne simple. Le correction_factor est recalculé à chaque nouveau résultat via recalibrateStrategy(). C'est de l'apprentissage en ligne sans gradient.
    incident_state : état transactionnel sérialisable, écrit et lu sous contrainte SELECT FOR UPDATE lors du claim multi-agent.
    agent_handoffs : diffusés via CDC changefeed → SSE → dashboard. Pas de polling. Event-driven proprement.
    detectIncidentStorm() : jointure incident_vectors × incident_state avec fenêtre temporelle et seuil cosinus — détection de cascade en une seule requête SQL. Quand ≥3 incidents similaires arrivent dans 10 minutes, PENDING_APPROVAL est forcé. C'est une décision d'architecture saine : un agent autonome qui relance tous les services ECS pendant une panne VPC amplifie la cascade.
    metric_snapshots : ingestion proactive pré-alarme. L'agent détecte les anomalies avant que CloudWatch tire.

L'embedding est maintenant sémantique. La chaîne Titan → Voyage → keyword est correctement implémentée et confirmée en production. La dégradation est gracieuse et non silencieuse — cliWarning est loggué quand le binaire échoue.

Ce qui coûte 1 point :

Le seuil de similarité CALIBRATION_THRESHOLD = 0.15 (distance cosinus < 0.15, soit similarité > 0.85) est très serré. Sur un test en production avec un incident CPU 92.5%, la distance retournée est 0.90 — ce qui tombe en fallback keyword. En base vide ou avec peu d'incidents historiques, le RAG ne se déclenche jamais. L'architecture est correcte mais le système ne montre pas sa pleine valeur avant que la incident_vectors soit substantiellement peuplée. Pour une démo devant jury, cela demande une préparation explicite.
2. Technical Implementation — 22 / 25

Ce qui est maintenant solide :

Le ccloud CLI s'exécute réellement. L'outil MCP execute_ccloud_command implémente une architecture deux couches cohérente — Layer 1 : execFile(CCLOUD_BINARY, args) avec timeout 15s, Layer 2 : REST API avec ccloudCommand renseigné pour traçabilité. Les deux couches retournent le même schéma de données — l'agent ne voit pas la différence. C'est du bon engineering.

L'intégration MCP officielle CockroachDB Cloud (cockroachlabs.cloud/mcp, transport StreamableHTTP/SSE, Bearer token) est réelle et non triviale. Le serveur MCP stdio maison expose 5 outils (execute_ccloud_command, aws_repair_service, crdb_cluster_health, crdb_list_slow_queries, crdb_query). La chaîne d'exécution callMcpTool() → McpClient → stdio subprocess est propre.

Les appels AWS réels sont documentés et fonctionnels :

    ECS : DescribeServicesCommand + UpdateServiceCommand(forceNewDeployment:true) + ListServicesCommand (auto-découverte)
    RDS : DescribeDBInstancesCommand + GetMetricStatisticsCommand (CloudWatch connections)
    Lambda : GetFunctionConcurrencyCommand + PutFunctionConcurrencyCommand

Le bug ?? 1000 Lambda — qui aurait plafondé une fonction illimitée à 1500 concurrences, l'aggravant au lieu de la réparer — est corrigé. ModifyDBInstance avec un parameter group inexistant est supprimé et remplacé par une recommandation manuelle avec approvalRequired: true. Ce sont des corrections d'ingénierie sérieuses, pas cosmétiques.

Le guard prompt-injection avant transmission à l'agent (webhook.ts) est rare dans un hackathon. Cela compte.

Ce qui coûte 3 points :

Premier point : le LLM de raisonnement en production est Anthropic direct API, pas Bedrock. Le boot log le confirme : "AI: anthropic 🟢 LIVE (direct API key)". Le code supporte Bedrock via AI_PROVIDER=bedrock et invokeBedrockThought() est implémenté — mais l'ECS task definition active Anthropic. Bedrock est utilisé pour les embeddings Titan (qui échoue et passe à Voyage), mais l'intelligence de l'agent passe par Anthropic. Ce n'est pas éliminatoire — ECS, SNS, CloudWatch, ECR, S3 sont tous des services AWS utilisés réellement — mais un jury attentif notera que le modèle de raisonnement n'est pas sur Bedrock.

Deuxième point : le endpoint /api/internal/cdc n'a toujours aucune authentification. La justification (CockroachDB changefeed ne peut pas envoyer de headers custom) est correcte. Mais la mitigation existe et n'a pas été appliquée : token query-param ?token=... validé côté serveur, ou whitelist IP. Une surface d'injection d'événements non mitigée dans un système qui déclenche des réparations AWS est un défaut de sécurité notable.

Troisième point : le [CCLOUD-BOOT] fetch status=404 visible dans chaque démarrage. L'appel REST pour récupérer les métadonnées d'org échoue systématiquement. Le système continue avec des données hardcodées. Ce n'est pas bloquant mais un juge qui ouvre les logs voit une erreur 404 au boot — cela demande une explication.
3. Real-World Impact — 23 / 25

Ce qui reste fort :

La boucle complète est démontrée et réelle : CloudWatch alarm → SNS → webhook → agent (diagnostician → remediator → auditor) → exécution MCP → feedback win-rate → calibration. Chaque maillon existe en production.

L'ingestion proactive de métriques (POST /metrics/ingest) est un différenciateur sérieux : l'agent détecte les anomalies avant que CloudWatch tire. L'incident CPU 92.5% ouvert automatiquement en réponse au test de Voyage AI le confirme. Le test ne visait pas à déclencher un incident — il l'a fait quand même.

La compression MTTR de 43 minutes (industrie P1) à quelques secondes de diagnostic est un argument quantifié et sourcé. C'est le bon niveau de pragmatisme pour un jury.

Ce qui coûte 2 points :

La boucle de calibration est architecturalement correcte mais le prior neutre de 0.5 sur les nouvelles stratégies maintient le système en PENDING_APPROVAL permanent jusqu'à accumulation de signal. Pour une démo de hackathon, si la base est vide, l'agent ne prend jamais de décision autonome. Le jury pourrait poser la question directement. La réponse honnête est "c'est un feature, pas un bug" — le système ne prend pas de risques sans preuve — mais elle doit être préparée et articulée.
4. Production Readiness — 18 / 20

Ce qui s'est amélioré depuis le verdict précédent :

    ECS_DEFAULT_CLUSTER corrigé (prod-cluster → cloud-surgeon) dans la task definition ET dans le code. Le bug silencieux qui rendait aws_repair_service inutilisable en production est résolu.
    VOYAGE_API_KEY injecté proprement dans AWS Secrets Manager avec le bon format ARN pour ECS. La procédure de gestion des secrets est correcte.
    Les outils de réparation dégradent gracieusement : auto-découverte ECS services, auto-découverte RDS instances, gestion du cas Lambda "pas de limite réservée" (évite de restreindre une fonction illimitée).
    CDC changefeed réutilisé sans recréation à chaque démarrage — comportement résilient documenté et fonctionnel.

Ce qui coûte 2 points :

Premier point : CDC endpoint non authentifié, mitigation non appliquée (voir Critère 2).

Deuxième point : pas de retry / dead-letter sur la chaîne SNS → agent. Si l'agent crash pendant le traitement, l'alarme est perdue. Ce n'est pas une opinion — c'est une garantie de perte d'événements sur défaillance transitoire. Une DLQ SNS avec re-drive est standard.
5. Creativity & Originality — 14 / 15

Inchangé depuis le verdict précédent. La boucle de calibration win-rate (bandit contextuel avec correction factor bayésien) reste le moment le plus original. L'idée qu'une base de données transactionnelle serve simultanément de mémoire vectorielle, de CDC event bus, et de ledger de win-rates — dans un seul moteur sans ETL — est une démonstration architecturale cohérente de ce que CockroachDB apporte.

Un point perdu : l'espace "agent auto-repair cloud" n'est pas vierge. Shoreline.io, PagerDuty Runbook, AWS Systems Manager Automation existent. La différenciation par CockroachDB CDC + vector memory + calibration loop est réelle mais doit être articulée en 90 secondes.

Verdict

Candidat sérieux à la 1ère place.

Les deux blocages techniques qui éliminaient la 1ère place sont résolus — ccloud binary en production, embeddings sémantiques confirmés. L'écart avec un score parfait tient maintenant à une seule chose opérationnelle :

Le RAG ne se montrera pas seul le jour de la démo.

Le système fonctionne correctement. Mais avec une base incident_vectors vide ou peu peuplée, chaque incident a une distance cosinus > 0.15 et passe en fallback keyword. Le jury posera la question standard : "montrez-moi un cas où la mémoire vectorielle a changé la décision". Si vous lancez POST /api/metrics/ingest en direct sans préparation, la réponse sera detectionMethod: "keyword" et ragScore: 0.90 — et l'architecture mémoire ressemble à un hasher sophistiqué.

Ce qu'il faut faire avant la soumission :

    Seeder incident_vectors avec 8 à 12 incidents résolus couvrant les stratégies principales (ecs_restart, rds_cpu_throttle, etc.), avec outcome_success: true et des embeddings Voyage réels.
    Préparer un scénario de démo où un nouvel incident CPU arrive, ragScore retourne < 0.15, et les logs montrent ragStrategyHint: "ecs_restart" dans le contexte — prouvant que la stratégie choisie vient du recall.
    Avoir la réponse à "pourquoi Anthropic et pas Bedrock pour le raisonnement ?" — la réponse correcte est "Bedrock est utilisé pour les embeddings Titan, Anthropic est le provider de raisonnement ; les deux sont commutables via AI_PROVIDER et le code Bedrock existe" — mais elle doit être dite avant que la question soit posée.

Ces trois points sont de la préparation de démo, pas de l'architecture. L'architecture est là.
